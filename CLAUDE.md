# LimeCloud 공통 작업 원칙

## 코드 작업 원칙
- 하드코딩 최소화
- 데이터는 DB 중심으로 유기적으로 연결
- 엔티티 간 관계를 명확히 설계하고, 화면에 필요한 데이터도 조합 가능한 구조로 만들 것
- 설정값, 상태값, 확장 가능한 목록은 테이블/enum/설정 구조로 관리, 코드 상수 하드코딩 남발 금지
- 코드 재사용 최대화, 불필요한 중복 구조 금지
- 확장 가능성은 확보하되 초기 구현은 과도하게 복잡하게 만들지 말 것
- 화면, DB, 스토리지, 승인 흐름, 알림 흐름이 서로 자연스럽게 연결되도록 설계

## 파일 열람 원칙
- 토큰 사용량 최소화
- 전체 파일을 불필요하게 길게 읽지 말고, 필요한 파일만 열람
- json 파일은 해당 작업과 직접 관련된 파일만 열람
- 수정 시 전체 파일을 다시 출력하지 말고, 변경이 필요한 부분만 확인 후 수정
- 문제 분석 시 에러가 난 위치와 직접 연결된 파일부터 최소 범위로 검토

## 프로젝트 핵심 전제
- Next.js App Router + Supabase
- Supabase Auth (세션 30일)
- Supabase Realtime 필요한 영역에 적용
- 2인 전용 커플 웹앱 (계정: lime / cloud)
- 대부분의 주요 변경은 상대 승인 필요
- 브레인스토밍은 승인 없이 즉시 반영
- 소프트 삭제 기본
- 이미지 표시용 1MB 이하 인코딩

## 승인 상태 (공통)
- draft(임시저장) → pending(승인대기) → approved(승인완료)
- pending → rejected(반려) → 즉시 draft 전환 + 히스토리 기록
- pending → cancelled(취소, 요청자만 가능)
- 동일 항목 중복 승인 요청 불가
- 삭제/비활성화/사진교체도 승인 필요
- 반려 시 반려 의견 필수

## 에이전트 역할 분담
- limecloud-pm: 총괄 설계/PM
- limecloud-backend: DB/API/Supabase
- limecloud-frontend: 페이지/컴포넌트
- limecloud-ui: 디자인 시스템/스타일
- limecloud-reviewer: 검증/리뷰
